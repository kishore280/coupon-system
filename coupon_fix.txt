COUPON SYSTEM - FIX LIST
========================

1. [FIXED 2026-07-14] site_url was mandatory in redeem() and reverse_redeem()
   (coupon_system/api.py) but only implicitly - no default value in the signature, and
   unlike phone/amount there was no explicit validation check.
   - If a caller omitted it, Python raised TypeError before the try/except could catch
     it, so it fell through as an unhandled 500 instead of the clean
     {"success": False, "error": "..."} shape every other failure returns.
   Fix applied: explicit `if not X or not isinstance(X, str) or not X.strip(): throw`
   checks added for phone, site_url, and invoice_no at the top of both functions.
   Still open: site_url is still trusted as-is once validated as a non-empty string -
   not derived from auth or checked against a known-branch registry. Anyone with a
   valid API key can claim any site_url.

2. [FIXED 2026-07-14] Type-confusion filter-injection risk in redeem()/reverse_redeem().
   Both pass invoice_no/site_url straight into dict-style filters
   (`frappe.db.exists("Coupon Ledger", {"invoice_no": invoice_no, "site_url": site_url,
   "type": "DEBIT"})`). Frappe's dict filters interpret a list value as
   [operator, value] - so a caller sending site_url as a JSON array (e.g.
   ["!=", ""]) instead of a string would pass the old `not site_url` check (a
   non-empty list is truthy) and get parsed as a filter *operator*, silently
   changing what the query matches (e.g. bypassing the "already redeemed" duplicate
   check). Fix applied: added `isinstance(X, str)` to the validation for phone,
   site_url, invoice_no in both functions.
   Still open (not yet audited/fixed): the same dict-filter pattern exists elsewhere
   in api.py with unvalidated params - e.g. `code` in scan()/redeem()
   (`frappe.db.get_value("Coupon Card", {"code": code}, "name")`). A full pass over
   every @frappe.whitelist method in api.py for this pattern is recommended.

3. [ADDED 2026-07-14] Store-restriction feature for redeem() only (by design, not scan()):
   - New `Coupon Store` doctype: store_name, site_url, is_active.
   - New `Coupon Campaign Store` child doctype backing a Table MultiSelect field
     `allowed_stores` on Coupon Campaign.
   - Empty allowed_stores = unrestricted (default, backward compatible with every
     existing campaign/card).
   - Enforced live (not snapshotted onto Coupon Card) via
     _get_campaign_allowed_site_urls() inside _assert_card_scannable(card, site_url=...),
     called only from redeem()'s code-based branch - scan() still passes no site_url
     so the earn flow is never store-gated.
   - Scope limitation: this only gates which store can redeem a specific card code.
     Once points are credited to a phone's balance they are fungible and can be spent
     anywhere via redeem()'s no-code balance-only branch - store restriction does not
     follow the points after they're earned.
   - [DEPLOYED 2026-07-14] Applied via `bench --site oxifixprivateltd.com migrate`,
     Oxifixprivateltd site process restarted. 8 new tests added to test_coupon_card.py
     (missing/type-confused site_url, unrestricted-any-store, correct-store-allowed,
     wrong-store-blocked, live-restriction-change, balance-redeem-ignores-restriction).
     Full coupon_system test suite (54 tests across all test files) passes:
     `bench --site oxifixprivateltd.com run-tests --module <module> --skip-before-tests`
     (--skip-before-tests needed - unrelated india_compliance/erpnext fixture gaps on
     this site break the default before_tests hook; not caused by this change).

5. [ADDED 2026-07-14] Query-count regression test for _get_campaign_allowed_site_urls
   (coupon_system/api.py). Measured empirically via a scratch script before writing the
   test: a full redeem(code=...) call is 54 total queries, only 3 of which are our new
   store-check code - the rest is Frappe's own per-document-save hook overhead
   (WhatsApp Notification, Workflow, Assignment Rule, Milestone Tracker checks fired by
   .insert()/.save(), unrelated to this feature). A blanket assertQueryCount on the
   whole redeem() call would be too brittle (any unrelated hook change elsewhere would
   break it). Instead added a targeted test asserting _get_campaign_allowed_site_urls
   itself stays at a fixed <=6 queries (2 batched lookups + information_schema
   column-cache misses on first access) even with 10 stores on a campaign - proves it's
   O(1) per redeem call, not O(N) per allowed store (it uses a single `name IN (...)`
   batch query, not a loop).

6. [ADDED 2026-07-14] Extracted _post_ledger(phone, entry_type, points, description,
   site_url=None, invoice_no=None) - the one place that writes a Coupon Ledger row.
   scan(), both redeem() branches, and reverse_redeem() each hand-rolled an identical
   frappe.new_doc("Coupon Ledger")...insert() block (4 copies). site_url/invoice_no were
   never required by the doctype itself (plain optional fields) - only redeem()'s own
   input validation requires them, which is correct for a store transaction but had
   gotten accidentally coupled to the ledger-writing mechanism itself. Separates "how to
   record a ledger entry" (mechanism, no site_url needed) from "what a given caller
   requires before calling it" (policy, e.g. redeem() requires site_url, withdrawal
   doesn't). No behavior change - full 56-test suite passed unmodified before deploy.

7. [ADDED 2026-07-14] Coupon Withdrawal Request - cash-out flow, built on _post_ledger.
   Researched against real precedent before building (ERPNext's own source + industry
   pattern research) rather than guessing:
   - New `Coupon Withdrawal Request` doctype: phone, points, amount (snapshotted),
     payout_details, status (Pending/Paid/Rejected), requested_on/paid_on/rejected_on,
     rejection_reason. Staff-only (System Manager/Coupon Manager) - NOT open to
     "Coupon Mobile" role, see open item below.
   - New `enable_withdrawals` (Check, default 0 - kill switch, opt-in) and
     `points_to_currency_rate` (Float, default 1) fields on Coupon System Settings.
   - request_withdrawal(phone, points, payout_details): checks enable_withdrawals first,
     validates inputs, locks the Coupon User row (frappe.get_doc(..., for_update=True))
     before checking available balance so two concurrent requests from the same phone
     can't both pass the same balance check, creates a Pending request.
     NO ledger entry is written at request time - a Pending request only locks points
     (excluded from future available-balance checks), it doesn't spend them. Mirrors the
     industry-standard available-vs-locked-balance pattern (confirmed against ERPNext's
     own Bin doctype: actual_qty - reserved_qty* = projected_qty, same shape).
   - _get_locked_withdrawal_points(phone)/_get_available_balance(phone): available =
     ledger balance - sum(points on that phone's Pending requests only). Must filter
     status=Pending specifically - Paid is already in the ledger (would double-count if
     included), Rejected never held anything (would wrongly and permanently shrink
     available balance if included).
   - Deliberately did NOT copy ERPNext's stored/incrementally-maintained reserved_qty
     counter - that exists in ERPNext because Bin is read/written across many linked
     documents at real scale where a live SUM would be a genuine perf problem. Our
     per-phone Pending-request count is 0-1 rows in practice - a live SUM is trivially
     cheap, and a stored counter would only add cache-invalidation risk for no benefit.
   - The actual DEBIT is written exactly once, by Coupon Withdrawal Request's own
     validate()/_handle_status_transition(), only when staff transitions Pending->Paid
     via a normal doc.save() in Desk. Rejected needs zero cleanup code - nothing was
     ever written to the ledger, so there's nothing to reverse (this is why Pending
     holds live in their own doctype instead of as a pseudo-state inside Coupon Ledger -
     keeps the ledger's one invariant, "only real finished facts", intact).
   - Double-Paid / double-debit race: verified (not assumed) against Frappe's own
     frappe/model/document.py - check_if_latest() (called from _save(), document.py:408)
     runs BEFORE run_before_save_methods()/validate() (document.py:414), and re-reads
     modified fresh from the DB each call (load_doc_before_save), raising
     TimestampMismatchError on a stale save. So two concurrent "mark Paid" saves on the
     same request can't both reach validate() (and thus _post_ledger) - the loser is
     blocked by Frappe's own optimistic concurrency before the side effect runs. No
     custom for_update lock needed on the Paid transition itself, only on request
     creation (see above), because request creation isn't a doc.save() race on an
     existing row, it's a check-then-insert race with no existing row to protect it.
   - amount is computed once at request creation from points_to_currency_rate and never
     recomputed on later saves (mirrors ERPNext Payment Entry's
     `if not self.source_exchange_rate: fetch_live()` snapshot idiom) - changing the
     global rate later never silently changes an already-created request's payout amount.
   - Tests added (test_coupon_withdrawal_request.py): disabled-by-default, success,
     insufficient balance, pending-locks-points-from-second-request, mark-Paid-writes-
     single-debit, mark-Rejected-has-no-ledger-effect-and-unlocks-points, cannot-change-
     status-of-non-pending, amount-not-recomputed-on-later-save, permission-denied.
   - [OPEN] Deliberately staff-mediated (Manager-only), not customer self-service via
     mobile, because request_withdrawal(phone, ...) takes phone as a free parameter with
     no binding to the caller's authenticated identity - the same pre-existing gap
     scan()/balance() already have (neither derives phone from the logged-in mobile
     session either). Letting "Coupon Mobile" call this directly today would let any
     mobile-authenticated user request a payout to their own bank details for someone
     else's phone number's points. Needs proper session-to-phone identity binding across
     the whole mobile-facing API before this can be safely opened to customers directly.

8. [ADDED 2026-07-14] Converted the Paid/Rejected transition to a real Frappe Workflow
   ("Coupon Withdrawal Request Approval", defined idempotently in code via
   ensure_withdrawal_workflow() in install.py, run from after_install/after_migrate - not
   manual Desk clicks, so it's reproducible and lives in git). Read Frappe's own
   frappe/model/workflow.py source before building rather than assuming behavior:
   - Reuses the doctype's own `status` Select field as workflow_state_field.
     create_custom_field_for_workflow_state() (workflow.py:42-61) only auto-creates a new
     hidden Link field `if not meta.get_field(self.workflow_state_field)` - since `status`
     already exists, it's reused as-is, confirmed empirically after migrate: 0 Custom
     Field rows created for this doctype.
   - Transitions: Pending -[Approve]-> Paid, Pending -[Reject]-> Rejected, both
     allowed="Coupon Manager", allow_self_approval=1 (this is a small-team tool, not a
     segregation-of-duties control - blocking self-approval would just be friction).
   - Workflow adds a role-gated permission/UI layer on top of the exact same doc.save()
     path already verified race-safe - it does NOT add its own locking
     (apply_workflow() -> doc.save()/submit()/cancel(), same check_if_latest() as
     everything else). Confirmed this doesn't break the existing 9 withdrawal tests
     (which call req.save() directly, not via apply_workflow()) because
     Administrator - what FrappeTestCase runs as - gets every Role automatically
     (frappe/permissions.py:506-507), so the Coupon-Manager-gated transition isn't
     blocked in tests. Our own controller's _handle_status_transition() (validate(),
     runs at document.py:414) still fires before Frappe's own validate_workflow()
     (_validate(), document.py:417), so our specific error messages ("Only a Pending
     request can change status") still win over Workflow's generic one for the same
     illegal-transition case.
   - Considered and initially recommended against Workflow (see the evaluation right
     before this entry) since it adds real setup machinery (Workflow + Workflow State +
     Workflow Transition + Workflow Action Master records) for zero extra concurrency
     safety over the plain-field approach - built it anyway per explicit request, but the
     underlying _post_ledger/lock/snapshot logic from point 7 is completely unchanged;
     Workflow only adds the approve/reject permission layer and Desk action buttons
     on top.
   - Full 65-test suite passes unmodified after adding the workflow. Deployed via
     bench migrate (creates the Workflow State "Paid" + the Workflow doc) + site
     restart.
   - [FOOTGUN, found via live testing on real committed data] apply_workflow(doc, action)
     (workflow.py:101-102) does `doc.load_from_db()` immediately on entry, which discards
     ANY in-memory field changes made before calling it - setting doc.rejection_reason
     right before calling apply_workflow(doc, "Reject") gets silently wiped, the field
     stays blank. apply_workflow only ever applies the state transition itself, never a
     simultaneous field edit. Practical guidance for staff: to record a rejection reason,
     do it as a SEPARATE save - either edit+save Rejection Reason first then click
     Reject, or click Reject first then edit+save the reason afterward (safe, since our
     controller only fires ledger/transition logic when status itself actually changes).
